Удаленные репозитории

Git позволяет работая в одном репозитории получать и принимать изменения из другого репозитория. В этом случае первый репозиторий называется локальным, а второй - удаленным.

Удаленный репозиторий может располагаться на том же компьютере, что и локальный, в локальной сети, или где то в интернете. Главное у пользователя должен быть доступ к нему одним из способов, который поддерживает git. Для получения изменений достаточно иметь доступ на чтение, для передачи изменений нужен доступ на запись.

Для того, чтобы находясь в локальном репозитории иметь возможность работать с удаленным репозиторием, надо подсоединить удаленный репозиторий к локальному, т.е. присвоить удалённому репозиторию имя (shortname) и указать его URL. Подсоединение выполняется командой git remote add <shortname> <url>. Эта команда сохраняет имя и URL удаленного репозитория в локальном конфигурационном файле.

Пусть у нас имеется репозиторий в текущей папке и репозиторий по пути /home/Thomas/project1, с которым мы хотим работать как с удаленным репозиторием. Так команда 

$ git remote add origin /home/Thomas/project1
подсоединяет репозиторий /home/Thomas/project1 как удаленный и назначает ему имя origin. В локальный конфигурационный файл добавится раздел????******************* 
[remote "origin"]
	url = ../gitguts6
	fetch = +refs/heads/*:refs/remotes/origin/*

Командой git remote без параметров можно посмотреть список всех подсоединенных удаленных репозиториев.
Эта же команда с параметром «-v» выводит дополнительно URL удаленных репозиториев.

Например: 

$ git remote
origin

$ git remote -v
origin	/home/Thomas/project1 (fetch)
origin	/home/Thomas/project1 (push)

Создание клонов репозитория. Команда git clone

Новый репозиторий можно склонировать, т.е. создать на основе существующего, командой git clone.
При клонировании репозитория происходит подсоединение клонированного репозитория как удаленного, с именем «origin».
При клонировании копируется текущая ветка (указатель текущей ветки) и она становится текущей веткой нового репозитория (обычно это ветка master).
Эта ветка настраивается как отслеживающая (см. долее) соответствующую удаленную ветку. ???? 

$ git clone /home/Thomas/project1 pr1_c

$ cd pr1_c

$ git remote
origin

Получение изменений из удаленного репозитория.
Удалённые указатели и удалённые ветки.
Команда git fetch

Команда git fetch загрузит с сервера все изменения, которых у вас еще нет.
Все новые объекты загружаются в локальную базу объектов .git/objects, новые указатели удаленного репозитория загружаются в каталог .git/refs/remotes/<имя_удаленного_репозитория>, а идентификаторы уже имеющихся там указателей обновляются.)
Эта команда не меняет содержимого вашего рабочего каталога.

Файлы каталога .git/refs/remotes/<имя_удаленного_репозитория>, являются копиями указателей удаленного репозитория и называются удаленными указателями. Если локальные указатели двигаются когда мы делаем коммиты в локальном репозитории, то удаленные указатели двигаются, когда мы получаем изменения из удаленного репозитория.

В каталоге .git/refs/remotes/<имя_удаленного_репозитория> помимо копий указателей удаленного репозитория хранится копия файла .git/HEAD *************????????????

Обращения к указателям (веткам) удаленного репозитория выглядят так: <имя_удаленного_репозитория>/<имя_указателя/ветки>. Например, если вы хотите посмотреть, как выглядела ветка master на сервере origin во время последнего соединения с ним, проверьте ветку origin/master. Если вы с партнёром работали над одной проблемой, и он выложил ветку iss53, у вас может быть своя локальная ветка iss53; но та ветка на сервере будет указывать на коммит в origin/iss53.

После получения изменений из удаленного репозитория можно выполнить слияние командой git merge. Только она изменит файла вашего рабочего каталога.

Допустим для получения обновлений из удаленного репозитория origin вы выполнили команду: 

$ git fetch origin
remote: Counting objects: 7, done.
remote: Compressing objects: 100% (2/2), done.
remote: Total 3 (delta 0), reused 3 (delta 0)
Unpacking objects: 100% (3/3), done.
From https://github.com/schacon/simplegit
 * [new branch]      serverfix    -> origin/serverfix
Из удаленного репозитория мы получили 7 новых объектов и новый указатель serverfix, как удалённый указатель origin/serverfix. Однако локальный указатель/ветка serverfix создан не будет!

Все удаленные указатели можно посмотреть командой git branch с опцией -r 

$ git branch -r
  origin/master
  origin/serverfix

Переключится на удаленную ветку можно так 

$ git checkout origin/serverfix

Вы можите слить удаленную ветку со своей текущей веткой, выполнив 

git merge origin/serverfix

Отслеживание веток

Есть команда работы с уделенными репозиториями, которые работают с т.н. отслеживаемыми ветками (upstream branch). Отслеживаемая ветка - это ветка удаленного репозитория, которая определенным образом связана с локальной веткой. Отслеживаемых веток может быть сколько угодно.

Создать локальную ветку на основе удаленной ветки и одновременно назначит удаленную ветку отслеживаемой можно командой git branch <ветка1> <удалённый_репозиторирий>/<ветка2>.

Например команда 

$ git branch fix origin/serverfix
Branch 'fix' set up to track remote branch 'serverfix' from origin.
создает ветку fix и назначает ей ветку origin/serverfix как отслеживаемую. После создания указателя fix, он указывают на тот же коммит, что и указатель origin/serverfix. 
$ git rev-parse origin/serverfix
28b529332be1d8bc8110bec50774d92d2e5d8f05

$ git rev-parse fix
28b529332be1d8bc8110bec50774d92d2e5d8f05

Информация об отслеживаемых ветках хранится в локальном конфигурационном файле. Так команда из предыдущего примера добавит в локальный конфигурационный файл такой «кусок» 

...
[branch "fix"]
	remote = origin
	merge = refs/heads/serverfix
...

Назначит текущей ветке отслеживаемую ветку можно командой git branch с ключом -u или --set-upstream-to.
Если вы хотите текущей ветке назначить отслеживаемую удаленную ветку <удалённый_репозиторирий>/<ветка>, то вызовите команду git branch -u <удалённый_репозиторирий>/<ветка>.
Той же командой можно переназначать отслеживаемые метки.

Создать ветку на основе удаленной ветке и сразу же переключиться на неё, можно командой git checkout --track <удалённый_репозиторирий>/<ветка>.
Эта команда создает локальную ветку с именем как у удаленной метки.
При этом созданная ветка, становится отслеживающей удаленную ветку на основе которой она была создана. 

$ git checkout --track origin/serverfix
Branch serverfix set up to track remote branch serverfix from origin.
Switched to a new branch 'serverfix'

Команда git checkout -b <ветка1> <удалённый_репозиторирий>/<ветка2> создает ветку с именем, отличным от имени удалённой ветки. Например: 

$ git checkout -b sf origin/serverfix
Branch sf set up to track remote branch serverfix from origin.
Switched to a new branch 'sf'
Теперь ваша локальная ветка sf будет отслеживать ветку origin/serverfix.

Посмотреть список локальных веток, с информацией о том какие ветки имеют удаленные отслеживаемые ветки, и на сколько коммитов локальные ветки опережает или отстает от локальной ветки, можно командой git branch с опцией -vv. Пример вывода: 

$ git branch -vv
  iss53     7e424c3 [origin/iss53: ahead 2] forgot the brackets
  master    1ae2a45 [origin/master] deploying index fix
* serverfix f8674d9 [teamone/server-fix-good: ahead 3, behind 1] this should do it
  testing   5ea463a trying something new

Видим что:
- ветка iss53 отслеживает ветку origin/iss53 и что локальная ветка опережает на два изменения, т.е. мы имеем два локальных коммита, которые не отправлены на сервер.
- ветка master отслеживает ветку origin/master и что локальная ветка в актуальном состоянии.
- ветка serverfix отслеживает ветку server-fix-good/teamone и что локальная ветка опережает на три коммита и отстает на один, т.е. есть один коммит на сервере, который мы еще не слили с веткой serverfix и три локальных коммита, которые вы еще не отправили.
- ветка testing не отслеживает удаленную ветку.

Эта команда не обращается к серверам. Данные об отставании или опережении основана на информации, которая получена при последнем подключении к удаленным репозиториям. Если вы хотите полностью быть в курсе опережающих и отстающих коммитов, вам необходимо извлечь данные из всех ваших удаленных репозиториев перед запуском этой команды. Например так: 

$ git fetch --all; git branch -vv

Если локальная ветка отслеживает ветку <удалённый_репозиторирий>/<ветка>, то в команде git ссылаться на эту отслеживаемую ветку можно с помощью конструкции @{upstream} или, что тоже самое @{u}.
Так, если вы находитесь на ветке master, которая отслеживает origin/master, то вместо git merge origin/master вы можите написать git merge @{u}.

Команда git pull

Команда git pull, по существу является эквивалентом двух последовательно выполняемых команд git fetch и git merge.
Так команда git pull <удалённый_репозиторирий> сначала получить все изменения с <удалённый_репозиторирий>, проделав тоже, что и git fetch <удалённый_репозиторирий>, а затем сольет текущую ветку с соответствующей отслеживаемой удаленной веткой.
Если у текущей ветки есть отслеживаемая удаленная ветка <удалённый_репозиторирий>/<ветка>, то она сольется с текущей, как по команде git merge <удалённый_репозиторирий>/<ветка>.
Если текущая ветка не отслеживает никакую удаленную, то операция слияния не выполняется.

Если находясь на ветке, у которой есть отслеживаемая удаленная ветка, вы наберёте git pull или git pull без указания удаленного репозитория, команда получит изменения из того репозитория, который входит в обозначение отслеживаемой удалённой ветке. Например если текущей веткой является ветка master и она отслеживает ветку origin/master, то изменения будут получены с репозитория origin.

Поскольку магия git pull может часто сбивать с толку, то обычно лучше явно использовать команды git fetch и git merge.

Отправка изменений. Команда git push

Когда вы хотите поделиться веткой с окружающими, вам необходимо отправить (push) её на удалённый сервер, на котором у вас есть права на запись. Не все ваши локальные ветки будут синхронизируются с удалёнными репозиториями. Вам нужно явно указать те ветки, которыми вы хотите поделиться. Так вы можете отправлять только те тематические ветки, над которыми вы работаете с кем-то совместно, и не отправлять свои личные ветки для работы, которую вы не хотите показывать.

Если у вас есть ветка serverfix, над которой вы хотите работать с кем-то ещё, вы можете отправить её точно так же, как вы отправляли вашу первую ветку.
Выполните git push <имя_удаленного_репозитория> <имя_ветки>: 

$ git push origin serverfix
Counting objects: 24, done.
Delta compression using up to 8 threads.
Compressing objects: 100% (15/15), done.
Writing objects: 100% (24/24), 1.91 KiB | 0 bytes/s, done.
Total 24 (delta 2), reused 0 (delta 0)
To https://github.com/schacon/simplegit
 * [new branch]      serverfix -> serverfix

Это в некотором роде сокращение. Git автоматически разворачивает имя ветки serverfix до refs/heads/serverfix:refs/heads/serverfix, что означает “возьми мою локальную ветку serverfix и обнови ей удалённую ветку serverfix.” Вы также можете выполнить git push origin serverfix:serverfix — произойдёт то же самое — здесь говорится “возьми мой serverfix и сделай его удалённым serverfix.” Можно использовать этот формат для отправки локальной ветки в удалённую ветку с другим именем. Если вы не хотите, чтобы ветка называлась serverfix на удалённом сервере, то вместо предыдущей команды выполните git push origin serverfix:awesomebranch. Так ваша локальная ветка serverfix отправится в ветку awesomebranch удалённого проекта.

Удаление веток на удалённом сервере

Скажем, вы и ваши соавторы закончили с нововведением и слили его в ветку master на удалённом сервере (или в какую-то другую ветку, где хранится стабильный код). Вы можете удалить ветку на удалённом сервере, используя опцию --delete для git push. Если вы хотите удалить ветку serverfix на сервере, выполните следующее:

$ git push origin --delete serverfix
To https://github.com/schacon/simplegit
 - [deleted]         serverfix

Сама команда удаляет только указатель на ветку на удаленном репозитории.
Как правило, Git-сервер оставит коммиты этой ветки на время, до тех пор пока не запустится сборщик мусора. Если ветка была удалена случайно, чаще всего ее легко восстановить.
